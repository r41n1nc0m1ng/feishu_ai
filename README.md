Openclaw Memory 功能需求文档
1. 需求背景
1.1  课题背景
- 在日常办公、研发协作和小团队项目推进过程中，大量关键决策并不会以正式文档的形式沉淀，而是散落在飞书群聊、临时讨论、会议纪要和任务沟通中。团队成员在推进项目时，经常会遇到以下问题：
  - 一是历史决策容易遗忘。团队曾经讨论并确定过某个方案，但几天或一周后再次遇到相关问题时，成员往往需要重新翻群聊、搜索关键词，甚至重新争论同一个问题。
  - 二是决策理由容易丢失。很多时候团队不仅需要知道“最终决定是什么”，还需要知道“当时为什么这么决定”“为什么没有采用另一个方案”。如果只记录结论、不记录理由，后续成员仍然难以理解上下文。
  - 三是决策版本容易混乱。小团队项目中经常出现“之前说 A，后来改成 B”的情况。如果没有版本更新机制，成员可能仍按照旧决策执行，造成沟通成本和执行错误。
- 因此，本项目将 OpenClaw 定位为飞书群聊中的“决策记录者”，围绕飞书群聊构建轻量决策记忆能力。系统不追求记录所有信息，而是聚焦于记录群聊中具有长期复用价值的决策、规则、约束、方案取舍和版本更新。
1.2 核心设计原则
- 本系统采用“群聊即记忆边界”的设计原则。每个飞书群被视为一个独立的 Chat Memory Space，机器人只记录其所在群中形成的决策信息，并默认只在该群中召回和推送。
- 采用群聊作为记忆边界的原因在于：对于小团队和高频协作场景而言，飞书群本身就是事实上的项目协作空间。谁在群里，谁就拥有该群上下文；如果某项决策需要同步给更多成员，团队通常会通过拉人进群或在群内统一发布完成同步。相比抽象项目空间、企业级记忆或个人记忆，群聊边界更自然、更轻量，也更容易控制权限和隐私风险。
- 系统不默认读取个人私聊和个人文档，不进行跨群自动共享，也不构建企业级全局记忆库。个人私聊只作为查询、确认和转发入口，不作为默认记忆来源。
1.3 目标用户
- 本产品主要面向以飞书群作为主要协作场所的小型团队。
- 这些团队通常具备以下特点：
  - 团队人数较少，一般为 3 至 10 人；决策过程较轻量，常常通过群聊直接协商；没有复杂的项目管理体系；历史决策容易散落在聊天记录中；成员需要快速确认“之前怎么定的”“为什么这么定”“后来有没有改”。

2. 功能体验链路
2.1 总体体验链路
1. 用户将机器人加入某个飞书群后，该群自动形成一个独立的 Chat Memory Space。机器人在群内监听消息，但不会把所有消息都写入记忆，而是先通过规则和模型判断消息是否可能包含高价值决策。
2. 当系统检测到群里出现决策、规则、约束、方案取舍或版本更新时，会生成候选记忆。对于高置信度、群内确认明显的信息，系统可以自动形成群聊决策卡片；对于不确定的信息，系统以轻提示方式询问是否记录。
3. 当用户后续在群里询问相关问题，或者群聊中再次出现与历史决策高度相关的话题时，系统从当前群的记忆库中检索相关决策，并以卡片形式返回结论、理由、来源和当前状态。
4. 如果后续群聊出现与旧决策冲突的新信息，系统触发矛盾更新判断，生成新版本记忆，并将旧版本标记为已覆盖。
机器人入群
  ↓
创建当前群的 Chat Memory Space
  ↓
群聊中出现讨论、结论、安排或变更
  ↓
候选记忆抽取
  ↓
系统进行信息分类
  ├── 时间安排 → 创建飞书日程
  ├── 明确执行事项 → 创建飞书待办
  ├── 长期决策信息 → 进入候选记忆判断
  └── 普通聊天 → 忽略或仅短期上下文缓存
  ↓
判断是否已有群内确认
  ├── 确认充分 → 写入群聊记忆并推送飞书卡片
  └── 确认不足 → 等待/询问
  ↓
后续群聊触发查询或语义相关讨论
  ↓
检索当前群记忆库
  ↓
返回历史决策卡片
  ↓
如出现冲突新决策 → 版本覆盖更新
2.2 决策形成与记录链路
- 群聊中的讨论不会在单条消息出现时立即形成正式记忆。系统需要观察一个短时间窗口内的群聊上下文，判断讨论是否已经形成相对稳定的一致结论，系统判断“一致决策形成”主要依据两个标准：
  - 第一，群内已有其他成员明确表示同意，并且在随后一段时间内没有成员提出反对、质疑或继续延伸该议题。这里的“一段时间”可以根据群聊活跃度设置为 3 至 10 分钟，或以若干条后续消息作为观察窗口。
  - 第二，群内已有其他成员明确表示同意，并且讨论自然切换到下一个决策议题或其他工作事项。这意味着上一个议题在群体互动中已经暂时收束，可以被视为阶段性结论。
- 例如群聊中出现如下内容：
A：我觉得这次不要做企业级记忆了，权限太复杂。
B：同意，先聚焦群聊决策记忆，Demo 更清楚。
C：OK，那白皮书也按这个方向写。
D：那我们接下来讨论 Benchmark 怎么设计。
- 系统检测到以下要素：存在明确方案取舍；存在理由；存在其他成员同意；没有成员继续反对；讨论已经切换到下一个议题。
- 此时系统生成“群内一致决策卡片”，并向群聊推送飞书卡片
- 卡片上保留删除选项，如果群成员认为机器人误判，可以点击“删除”按钮。若在反悔窗口内无人点击删除，则系统将该决策正式写入本群长期记忆。
- 这种机制的设计目的，是在不增加复杂审批流程的前提下，为自动记忆提供一个轻量纠错入口。系统默认尊重群聊共识，但允许群成员快速否决误记。
2.3 未形成一致决策的讨论进度记录链路
- 当群聊中出现明显的决策讨论，但没有形成一致结论时，系统不应强行记录为正式决策，而应记录当前讨论进度。
- 例如群聊中出现如下内容：
A：我们就别做企业级记忆了吧。
B：我觉得也可以，但要不要保留企业级作为远期规划？
C：这个还得再想想，可能答辩时会被问到。
- 系统判断该讨论具有项目价值，但尚未形成一致结论。此时机器人同样向群聊推送飞书卡片，但卡片类型不是“已形成决策”，而是“讨论进度记录”：
检测到本群正在讨论一个尚未收束的问题：

议题：是否完全放弃企业级记忆，或将其保留为远期规划。

当前进度：
- 倾向方案：MVP 阶段不做企业级记忆；
- 待确认问题：是否在白皮书中保留远期规划表述；
- 当前状态：尚未形成一致决策。

 [删除]
2.4 历史召回链路
- 一周后，群里有人问：我们之前为什么不做企业级记忆来着？
- 机器人从当前群的记忆库中检索相关内容，返回：
根据本群历史决策：

MVP 阶段暂不做企业级记忆，当前聚焦群聊决策记忆。

当时的理由是：企业级记忆会扩大权限、个人文档、私聊和跨群治理边界，不利于比赛 Demo 聚焦；而群聊本身已经提供了天然的协作边界。

状态：生效中
来源：4 月 24 日本群讨论
2.5 冲突更新链路
- 群里后续出现新的讨论：
A：之前说完全不做个人入口，但我觉得可以保留私聊查询入口。
B：同意，个人私聊只用于查询和确认，不作为默认记忆来源。
C：那就这样改。
- 系统检测到该内容与旧记忆存在部分冲突，于是生成新版本：
已更新一条群内决策：

新决策：保留个人私聊入口，但仅用于查询、确认和转发，不作为默认记忆来源。

被覆盖的旧表述：完全不做个人入口。

状态：新版本生效
来源：本群最新讨论
- 系统保留旧版本，但将其标记为 Deprecated，后续查询时优先返回新版本。
2.6 日程与待办分流链路
- 系统不把所有信息都作为长期记忆，对于明确时间安排或执行任务，系统优先调用飞书原生能力，创建待办或约会议，会影响未来判断的上下文才存入记忆。
2.7 个人私聊链路
- 个人私聊不是默认记忆来源，只作为辅助入口。

3. 功能体验情景
情景一：团队确定产品边界
- 场景背景：一个 5 人团队正在飞书群中讨论 Memory Engine 的产品边界。团队曾经考虑过企业级记忆、项目空间、个人记忆等方案，但发现范围过大，难以在比赛周期内落地。
- 群聊过程：
A：我觉得企业级记忆先别做了，太重。
B：对，涉及权限、跨群、个人文档，比赛 Demo 很难讲清楚。
C：那我们就按群聊存决策吧，一个群就是一个记忆空间。
D：这个更符合小团队场景，也更容易测抗干扰和冲突更新。
A：OK，那就定这个方向。
- 系统行为：系统识别到这是关于产品边界的重要决策，且有多人确认，因此生成群内决策记忆。
- 价值：团队后续不需要反复讨论“为什么不做企业级记忆”，系统可以直接召回历史理由。
情景二：群聊中出现大量无关信息后仍能召回
- 场景背景：团队在确定产品边界后一周内继续在群里讨论 UI、答辩、代码问题和日程安排。大量无关消息淹没了原始决策。
- 用户提问：
我们之前到底为什么说不要做项目空间了？
- 系统行为：系统只在当前群的 Chat Memory Space 中检索，并优先召回最新版本的相关决策。
- 价值：系统证明自己不是简单关键词搜索，而是在大量干扰消息后仍能准确召回历史决策及其理由。
情景三：固定时间安排自动转日程
- 场景背景：团队在群里约定明天下午进行 Demo 评审。
- 群聊过程：
A：明天下午 3 点我们开一次 Demo 评审会。
B：可以。
C：我也可以。
- 系统行为：系统识别这是固定时间安排，不进入长期决策记忆，而是询问是否创建飞书日程。
- 价值：系统将执行事项交给飞书原生工具，不污染长期记忆库。
情景四：群里有人问一个已有答案的问题
- 场景背景：新加入群的成员不了解此前讨论。
- 用户提问：
为什么我们不用项目空间，而是按群存？
- 系统行为：机器人自动召回相关历史决策，并用简洁口语化方式回答。
- 价值：新成员可以快速理解历史上下文，老成员不必重复解释。

4.1 系统模块拆解

  ↓

  ↓

  ↓

  ↓

4.2  飞书接入层
- 目标：接收机器人所在群的消息事件，并支持机器人向群内发送卡片消息、确认按钮和历史决策回复。
- 输入：飞书群消息事件，包括文本消息、@机器人消息、群成员回复、机器人按钮点击事件等。
- 处理逻辑：系统接收消息后，首先读取 chat_id，并将该消息路由到对应的 Chat Memory Space。系统不处理机器人未加入群的消息，也不默认读取个人私聊和个人文档。
- 输出：
{ 
  "message_id": "om_x100b518cd8a88cb8b20d0d347927531", 
  "sender_id": "ou_17835e9955b265dd30ca56e627d3a6ca", 
  "chat_id": "oc_b42f6cfaaf70a03a024f4bc0816fb2bf", 
  "chat_type": "group", 
  "timestamp": "1745506614", 
  "group_subject": "oc_b42f6cfaaf70a03a024f4bc0816fb2bf", 
  "text": "消息本身的上下文调用的是什么接口..." 
}
- 目前openclaw获取信息的逻辑是，群聊中每发一条消息，就会传到openclaw，似乎目前已经对不同的群聊做了分区，不同消息来源不会让openclaw“串台”，但是似乎只是开了不同的会话
- openclaw能够查询群聊最近的几条消息

代：context同时包含了：进来的新消息和memory注入
如果所有的context都会留在该群的一个session内部。长期下来确实会有上下文问题。
这样的策略兼容性比较好：保留一个轻量总入口视角：hook+主session，再把消息路由到少数活跃 topic contexts
 一个可行实现：让路由层只维护“当前分发所需的最小视图”：短期时间窗该群消息 + memory hints + 事件/议题元信息。这样旧消息不会长期以原始上下文形式干扰后续分发；持续有影响的信息则通过 memory 或状态摘要保留下来。
后续可以边实现边论证这个。这些要根据设计理念、目标效果和后续落地的可行性 来设计。

4.3 消息预处理层
- 目标：减少无关消息进入openclaw占用上下文，减少调用次数，降低成本和噪声。
宋：定量会不会截断了讨论。我感觉1➕2会好点。
代：可能的方案：“和上述策略相关 路由层负责联通，积累一定讨论进度or达到定时后分发给后续的事件层。”
- 这里存在一个需要论证的地方，如何让系统在“看到聊天信息时”保持静默，直到讨论完毕形成决策
  - 1、关键词触发+轻量LLM判断，通过事件订阅实时接收群消息，后端检索到关键词如“计划”“决定”等开始储存消息，然后发给LLM节点判断语义，抉择是否需要发给openclaw让他开始工作
  - 2、定时扫描，每隔一段时间抓聊天记录，调用LLM节点扫一下有没有决策或直接发给openclaw
  - 3、定量扫描，写一个机器人监听，每达多少条数据就调用LLM节点扫一下有没有决策或直接发给openclaw
  - 4、只开启@功能，手动@后开始总结存入记忆，繁琐
4.4 openclaw内部技能选择层
- 目标：判断一条信息应该进入哪条处理链路。
- 分类结果：
  - Schedule：固定时间安排
Task：明确执行事项
Memory：长期群聊记忆
Ignore：普通聊天或无价值信息
- 分类标准：
  - 如果信息包含明确时间、会议主题和参与对象，则优先识别为 Schedule。
  - 如果信息包含负责人、任务动作和截止时间，则优先识别为 Task。
  - 如果信息包含方案取舍、决策结论、规则约定、约束边界、版本变更、历史理由，则识别为 Memory。
  - 如果信息不具备上述特征，则忽略或仅短期缓存。
- 示例：
“明天下午 3 点开评审会” → Schedule
“张三周五前完成接口联调” → Task
“本期不做企业级记忆，聚焦群聊决策记忆” → Memory
“收到”“哈哈哈”“我晚点看” → Ignore

4.5 结果形成与确认推送层
- 目标：判断群聊中的讨论是否已经形成一致决策，并通过飞书卡片完成轻量确认或纠错。
- 一致决策形成标准：
  - 标准一：群里至少有一名其他成员明确表示同意，并且在观察窗口内没有其他成员反对、质疑或继续延伸该议题。
  - 标准二：群里至少有一名其他成员明确表示同意，并且群聊自然切换到下一个决策讨论、任务安排或其他工作议题。
- 同意信号：同意、OK、可以、就这样、按这个来、确定、收到，那就这么办、我同步到文档里、那就按这个写、没问题
- 反对或延伸信号：等等、还没定、不同意、再讨论、先别、这个只是想法、不一定、有没有必要、再想想、还有一个问题、那如果……
- 议题切换信号：开始讨论下一个方案、进入任务分配、开始讨论日程、开始讨论 Benchmark、开始讨论 UI、开始讨论代码实现、出现明显无关的新议题
- 当然这部分openclaw的能力应该足够判断，或者也仅仅通过Agent节点的形式
- 决策卡片流程：无论一致决策是否形成都将总结推送到飞书卡片，若未达成一致需注明
- 状态规则
  - 如果只有单人表达，没有确认信号，要注明表达了什么，系统可以根据内容价值生成讨论进度卡片。
  - 飞书卡片要添加删除按钮，默认储存、点击则删除
  - 如果多人确认，且一段时间内没有反对或继续延伸，则进入一致决策卡片流程。
  - 如果多人确认，且群聊已经切换到下一个决策讨论或工作议题，则进入一致决策卡片流程。
  - 如果新记忆与旧记忆冲突，则进入冲突更新流程。

4.7 记忆检索与触发层
4.7.1 触发方式
- 群内 @机器人提问
  - 当用户在群内 @机器人提问时，系统只检索当前 chat_id 下的记忆
  - 系统返回当前群中的相关历史决策、理由、状态和来源等。
- 群内相似疑问自动触发
  - 当群聊中出现与历史记忆高度相似的话题，且表达形式为疑问、确认或追问时，系统可以主动召回历史记忆。
  - 示例：
我们是不是也可以做一个企业级记忆？
之前为什么没做来着？
这个是不是已经讨论过？
  - 系统判断该问题与已有记忆高度相关后，主动推送历史决策卡片。若只是普通陈述或新的开放讨论，系统不应频繁打断。
- 这个面临前面同样的问题，如果想要机器人及时回复，似乎只能通过实时LLM节点判断
- 用户私聊查询
- 这个要论证下好不好做，因为涉及到用户鉴权，不好做就先不做了
4.7.2 检索方式
- 采用混合检索：关键词检索 + 向量语义检索 + 状态过滤。
- 优先级如下：
  - 新版本 > 旧版本
  - 高相似度 > 低相似度
  - 有明确理由和来源的记忆 > 信息不完整的记忆
4.8 冲突更新
- 触发条件：新候选记忆与已有 Active 记忆在同一主题下存在冲突或修正关系。
- 流程：
识别新记忆
  ↓
检索同主题旧记忆
  ↓
判断是否冲突或覆盖
  ↓
生成冲突提示
  ↓
群内确认
  ↓
旧记忆标记 Deprecated
  ↓
建立 supersedes 版本链
4.9  权限与隐私流程
- 系统采用最小权限原则：机器人只处理自己所在群的消息；每个群的记忆仅属于该群；不同群之间默认不共享；个人私聊不作为默认记忆来源；个人文档不进入本期范围；机器人回复时不暴露未公开来源。
